# Backlog — Mogura

現行コードベースをレビューしたうえでのリファクタリング案・機能追加案。優先度は主観。

## リファクタリング案

### R1. `main.go` のオーケストレーション層抽出（優先度: 高）
現状 `main.go` が flag 解析 / scan / analyzer 呼び出し / waste 合成 / savings 計算 / 出力分岐まで全部抱えている（約 115 行）。`internal/app` などに `Run(args, stdout, stderr) int` を切り出して、`main.go` はエントリだけにする。テスト容易性も上がる。

### R2. 「分析結果」集約型の導入（優先度: 高）
`main.go` で `files` から 7 種類の analyzer を個別に呼び出し、JSON/テキスト分岐で同じものを再構築している。`analyzer.Analyze(files, opts) → Result` のように一度にまとめて返す集約型を作ると、formatter は `Result` だけ受け取れば良くなり、分岐の二重管理が消える。

### R3. `DetectWaste` と `DetectLargeGitDirs` の統合（優先度: 中）
両者とも「ディレクトリパターンに一致する祖先パスの size を畳み上げる」という同じ構造。`detectAncestorDirs(files, matcher)` 的な共通関数に寄せられる。現状は `.git` だけ別実装になっていて、パターン追加のたびにロジックが増えがち。

### R4. Scanner の並列化（優先度: 中）
`filepath.WalkDir` は単一 goroutine で走査。大規模ディレクトリでは I/O 待ちがボトルネックになりうる。ディレクトリ単位で worker pool に投入する実装に変えると数倍速くなる可能性（要ベンチ）。ただし外部依存ゼロは維持。

### R5. `FormatSize` の精度制御オプション化（優先度: 低）
現状は常に小数 1 桁固定。JSON では生バイトがあるので問題ないが、テキスト出力で「4 B」と「4.0 KB」が混在して読みづらい箇所がある。`FormatSizeWith(bytes, precision)` を追加して表示側の一貫性を整える。

### R6. `wastePatterns` の外部化（優先度: 低）
現状 map リテラルにハードコード。ユーザーが `~/.mogurarc` 等で追加できるようにする下地として、`DefaultWastePatterns()` を関数化し、`DetectWasteWith(files, patterns)` を受けられる形にしておく。

### R7. `formatter/table.go` のテンプレ共通化（優先度: 低）
`PrintDirTable` / `PrintExtTable` / `PrintCategoryTable` / `PrintTopFiles` が似たパディングロジックを各自持っている。`text/tabwriter` を使うか、小さな `renderTable(headers, rows)` ヘルパに寄せるとコード量が半減する。

### R8. `internal/analyzer` の責務が大きすぎる問題（優先度: 中）
7 ファイルがフラットに同居。`analyzer/size`（dir/ext/topn/tree）と `analyzer/insight`（waste/stale/category）にサブパッケージ化すると意味的な境界が明確になる。ただし依存ルールの維持に注意。

## 機能追加案

### F1. インタラクティブ TUI モード（優先度: 高）
`ncdu` 風に矢印キーでディレクトリをドリルダウンできる TUI。`-tui` フラグで起動。標準ライブラリのみだと制約が大きいので、この機能だけは外部依存（bubbletea 等）を許容するか別バイナリ化するか要検討。

### F2. 除外パターン（`-exclude`）（優先度: 高）
`.git` や特定ディレクトリをスキャン対象から除外するフラグ。`-exclude 'node_modules,*.tmp'` のように複数指定。`.gitignore` の読み込みも将来的にはあり得る。

### F3. 差分モード（`-diff`）（優先度: 高）
前回スキャン結果（JSON）と現在を比較して「どこが増えた/減ったか」を出す。`mogura -json > prev.json` → 後日 `mogura -diff prev.json .` でサイズ増加 Top N を可視化。容量肥大の原因追跡に強力。

### F4. 重複ファイル検出（優先度: 中）
サイズが同じファイルをグルーピングしてハッシュ比較（初段サイズ、次段先頭 4KB、最後に全体 SHA-256 の三段構え）。`-dup` フラグ。節約可能量が具体化する。

### F5. CSV / Markdown 出力（優先度: 中）
既存の JSON に加えて `-csv` / `-md` を追加。レポート共有用。formatter 層の拡張で済むので低コスト。

### F6. 進捗表示（優先度: 中）
大規模ディレクトリのスキャン中に stderr に進捗（N files scanned... / 現在のディレクトリ）を出す。`-quiet` で抑制。

### F7. フィルタ条件（`-min-size` / `-ext`）（優先度: 中）
`-min-size 10M` でそれ未満のファイルを集計から除外、`-ext mp4,mkv` で特定拡張子のみ対象、など。既存の analyzer を呼ぶ前段でスライスを絞るだけ。

### F8. 削除候補のインタラクティブ確認（優先度: 低）
waste / stale の結果に対して `-clean` フラグで一件ずつ y/n 確認して削除。※ CLAUDE.md の「削除機能は実装しない」方針に反するので要方針再検討。方針維持なら却下。

### F9. ベンチマーク / プロファイリング基盤（優先度: 低）
`cmd/bench` あるいは `-cpuprofile` / `-memprofile` フラグ。R4 の並列化や大規模対応で効果測定に必要になる。

### F10. ディレクトリサイズの履歴追跡（優先度: 低）
`~/.mogura/history/` に日次スナップショットを保存し、時系列グラフ（ASCII）で表示。長期的な傾向把握用。cron と組み合わせる。

### F11. Homebrew Tap 配布（優先度: 低）
`goreleaser` 等で `brew install mogura` を可能に。配布チャネル整備。

### F12. マウント境界の尊重（`-x` / `--one-file-system`）（優先度: 中）
`du -x` 相当。外付けドライブや network mount を跨がないオプション。Mac ユーザには実用的。

## 可視化・ビジュアライズ案（別枠・重点領域）

ターミナル内で完結する案と、ブラウザに出力する案の二系統。外部依存ゼロの制約下でどこまで「見て楽しい / 直感的」にできるかが鍵。

### V1. ASCII 棒グラフ（優先度: 高・低コスト）
カテゴリ別・拡張子別・ディレクトリ Top N の各テーブル行末に `████████░░░░` のような水平バーを添える。最大値を基準に正規化するだけで済み、標準ライブラリのみで実装可能。サマリ画面が一気に「分析ツールっぽく」なる。

例:
```
動画         42.3 GB  ████████████████████░░░░░  52.1%
画像         18.1 GB  ████████░░░░░░░░░░░░░░░░░  22.3%
コード        8.4 GB  ███░░░░░░░░░░░░░░░░░░░░░░  10.4%
```

### V2. ツリーマップ（Treemap）ASCII 版（優先度: 高・中コスト）
ブロックサイズに比例する矩形を端末幅に敷き詰める。Squarified Treemap アルゴリズムを Go で実装（200 行程度）。`disk inventory X` のような一枚絵がターミナルで見られる。ANSI 256 色でカテゴリ別に色分け。

```
┌───────────────┬───────┬─────┐
│               │       │     │
│  node_modules │ .git  │ src │
│               ├───────┤     │
│               │ dist  │     │
└───────────────┴───────┴─────┘
```

### V3. サンバースト / ドーナツ（ASCII）（優先度: 中）
階層を同心円で表現。ASCII では厳しいが、Unicode Braille（⠿ など）を使えば擬似的な円グラフは描ける。ツリー深さの可視化に向く。ただし端末フォント依存が強く、見栄えは博打。

### V4. ヒートマップカレンダー（優先度: 中・`ModTime` 活用）
ファイル数/サイズをアクセス時間軸に並べた GitHub 風のヒートマップ。「いつ作ったファイルが何 GB あるか」が一目で分かる。stale 検出の根拠も示せる。ASCII でも Unicode ブロック `░▒▓█` で濃淡表現可能。

```
2025  ▁▂▃▂▁▁░░░░▂▃▇█▇▃▁▁
       Jan           Dec
```

### V5. スパークライン（優先度: 中・F10 と連携）
履歴追跡機能（F10）とセットで、ディレクトリごとのサイズ推移を 1 行スパークライン `▁▂▃▅▆▇█` で表示。「どこが最近急増したか」が視覚的に分かる。

### V6. HTML レポート出力（`-html`）（優先度: 高・高コスト）
JSON からブラウザ用の単一 HTML ファイルを生成。D3.js を CDN 経由で読み込んで本格的な Treemap / Sunburst / Flame graph を描く。Go 側は `embed` で HTML テンプレを埋め込むだけなので外部依存ゼロは維持できる。一番インパクトが大きい可視化はこれ。共有もしやすい。

### V7. SVG 直接出力（`-svg`）（優先度: 中）
HTML より軽量。`encoding/xml` で SVG を直接生成し、Treemap や棒グラフを画像として出力。ブログ貼り付け・PR 添付に便利。依存ゼロ維持。

### V8. ターミナル画像プロトコル対応（優先度: 低）
iTerm2 / Kitty / wezterm は画像インライン表示に対応しているので、V7 の SVG を PNG 変換（ここは要外部依存）して直接ターミナルに描画。Mac 中心ユーザにはマッチするが画像変換ライブラリが要る。

### V9. Flame Graph 風「サイズ積層図」（優先度: 中）
横軸がサイズ、縦軸がディレクトリ階層。`perf` の Flame Graph と同じ発想でディスク使用を表現。大きなディレクトリが横に長く伸びるので直感的。ASCII / SVG 両対応可能。

### V10. カラー付き TUI ダッシュボード（F1 と統合）（優先度: 高）
F1 の TUI をベースに、左ペインにツリー、右ペインに選択ディレクトリの Treemap / 棒グラフをリアルタイム更新。これが最終形の「可視化体験」。ただし外部依存（bubbletea + lipgloss）が事実上必須。

## 可視化ロードマップ（推奨順）

1. **V1 (ASCII 棒グラフ)** — 1 日で済む。既存テーブルの印象を劇的に改善。まずこれ。
2. **V4 (ヒートマップ)** — `ModTime` 既に持っているので追加コスト低。stale 機能と相性抜群。
3. **V2 (Treemap ASCII)** — Squarified アルゴリズム実装が楽しい。mogura のアイコン的出力になる。
4. **V6 (HTML レポート)** — JSON があれば `embed` + D3 で高完成度。共有価値が一番高い。
5. **V10 (TUI ダッシュボード)** — 外部依存の方針を決めたうえで最終形として。

## 優先度サマリ

**リファクタ / 機能**: R1 → R2（土台整理）→ F2（除外パターン）→ F3（差分モード）→ R3 の順。
**可視化**: V1 → V4 → V2 → V6 → V10。低コストで見栄えが変わる V1 と V4 を先行、Treemap（V2/V6）で独自色を出す戦略。
