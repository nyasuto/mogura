タスク設計ガイドライン:

1. 粒度: 1 タスク = 読むべきファイルが 1〜3 個で済むサイズ

- Phase 完了時に次 Phase のタスクを展開する

## Backlog

<!--
Ralph Loop の実行中に気づいた「今のタスクではないが、いつかやるべきこと」をここに溜める。
Claude Code が作業中に追記し、人間が Phase 間のレビュー時にトリアージする。

分類:
- 🐛 bug:     発見したがスコープ外のバグ
- 💡 idea:    思いついた機能追加・改善アイデア
- 🔧 refactor: 動いているが改善したいコード
- 📝 doc:     ドキュメントの不足・改善
- ⚠️ debt:    技術的負債（仮実装、ハードコード等）

エントリの書式:
  - [分類] 内容（発見イテレーション or 日付）

例:
  - 💡 CLI に --verbose フラグを追加する（iter 12）
  - 🔧 parser.go の switchを map lookup に変える（2026-04-01）
  - ⚠️ エラーメッセージに位置情報が含まれていない箇所がある（iter 8）
  - 🐛 空文字列入力時に panic する（iter 15）

トリアージのルール:
  1. Phase 境界で人間が Backlog をレビューする
  2. 次 Phase に昇格するものは tasks.md のタスクとして `- [ ]` に変換
  3. やらないと判断したものは ~~取り消し線~~ で残すか削除
  4. 急ぎのバグは現在の Phase に差し込んでよい
-->
