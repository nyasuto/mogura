# PRD: Mogura — Mac ディスク使用量アナライザ

> **プロジェクト名**: Mogura（モグラ — ディスクの中を掘って調べる）
> **言語**: Go
> **開発手法**: Ralph Loop（Claude Code）
> **ステータス**: Draft v1.0
> **作成日**: 2026-04-13

---

## 1. 概要

macOS 標準のストレージ管理は「システム」「アプリケーション」「書類」などの大雑把なカテゴリしか示さず、具体的にどのディレクトリ・ファイルがディスクを圧迫しているのか把握しづらい。

Mogura は指定ディレクトリ以下を高速にスキャンし、ディレクトリ別・拡張子別・カテゴリ別のサイズ集計、巨大ファイルの発見、キャッシュ・不要ファイルの検出を行う Go 製 CLI ツール。

## 2. 設計原則

- **シングルバイナリ**: `go build` で単一バイナリを生成
- **ゼロ外部依存**: 標準ライブラリのみ
- **高速**: 並行ディレクトリウォーク（goroutine + channel）
- **人間に読みやすい出力**: サイズは自動で KB/MB/GB 表記、ツリー表示対応

## 3. アーキテクチャ

```
CLI (flags解析)
  → Scanner (ディレクトリウォーク + メタデータ収集)
  → Analyzer (集計・分類・ランキング)
  → Formatter (テーブル / ツリー / JSON 出力)
```

### 3.1 横断的設計（Phase 1 で確定）

- **FileInfo 型**: `Path`, `Size`, `ModTime`, `IsDir`, `Ext` を持つ統一型。全パイプラインでこの型を使う
- **サイズ表現**: 内部は常に `int64`（バイト）。表示時のみ human-readable に変換
- **エラー処理**: パーミッションエラー等は stderr に警告を出してスキップ。スキャン全体を止めない

## 4. フェーズ設計

### Phase 1: スキャン基盤

**目標**: 指定ディレクトリ以下の全ファイルサイズを集計して表示できる

| タスク | 完了基準 |
|--------|----------|
| プロジェクト初期化 | `go build ./...` が通る |
| FileInfo 型と human-readable サイズ変換 | テスト付きで定義済み |
| ディレクトリウォーカー（単一 goroutine） | 指定パス以下を再帰走査、FileInfo のスライスを返す |
| ディレクトリ別サイズ集計 | 各ディレクトリの合計サイズを計算 |
| CLI エントリポイント | `mogura /path` で合計サイズとディレクトリ上位10件を表示 |

### Phase 2: 分析機能

**目標**: 拡張子別・カテゴリ別の集計と巨大ファイル検出

| タスク | 完了基準 |
|--------|----------|
| 拡張子別サイズ集計 | `.mp4`, `.log` 等ごとの合計サイズ + ファイル数 |
| カテゴリ分類ルール定義 | 拡張子 → カテゴリ（動画/画像/コード/ドキュメント/キャッシュ/その他）のマッピング |
| カテゴリ別サイズ集計 | カテゴリごとの合計サイズ + 割合表示 |
| 巨大ファイル Top N | サイズ順で上位 N 件のファイルパスとサイズを表示（デフォルト20） |

### Phase 3: 出力フォーマット

**目標**: 複数の出力形式に対応

| タスク | 完了基準 |
|--------|----------|
| テーブルフォーマッタ | 列幅揃え・右寄せサイズのテキストテーブル |
| ツリーフォーマッタ | `tree` コマンド風のディレクトリツリー + サイズ（深さ制限付き） |
| JSON フォーマッタ | `--json` フラグで全結果を JSON 出力 |
| CLI フラグ統合 | `--top N`, `--depth N`, `--json`, `--sort size|name|count` |

### Phase 4: ゴミ発見

**目標**: 削除候補を提示する

| タスク | 完了基準 |
|--------|----------|
| 既知キャッシュディレクトリの検出 | `node_modules`, `.cache`, `__pycache__`, `DerivedData`, `.Trash` 等を検出・合計サイズ表示 |
| 巨大 .git ディレクトリの検出 | `.git` ディレクトリのサイズを個別表示 |
| 古いファイルの検出 | 最終更新が N 日以上前のファイルを検出（`--older-than 365` 等） |
| サマリレポート | 全分析結果の統合サマリ（総容量、カテゴリ内訳、削除候補、節約可能量） |

### 5.2 イテレーション設計

- 1 イテレーション = 1 タスク（tasks.md の 1 チェックボックス）
- Phase 完了時に次 Phase のタスクを tasks.md に展開

### 5.3 Backlog 運用

- Claude Code がタスク実行中に気づいた改善点は `## Backlog` に分類タグ付きで追記
- Backlog のアイテムを Claude Code が勝手にタスク化・実装することは禁止
- Phase 境界で人間がトリアージ

### 5.4 人間の介入ポイント

- Phase 境界での Backlog トリアージ
- カテゴリ分類ルールのレビュー（Phase 2）
- 既知キャッシュディレクトリのリスト確認（Phase 4）

## 6. リスクと軽減策

| リスク | 影響 | 軽減策 |
|--------|------|--------|
| シンボリックリンクの無限ループ | スキャン停止 | リンクを検出したらスキップ、visited set で二重カウント防止 |
| パーミッションエラー | 一部ディレクトリが読めない | stderr に警告、スキャンは継続 |
| 巨大ファイルシステム（数百万ファイル） | メモリ圧迫 | Phase 1 ではストリーミング集計で全 FileInfo を保持しない設計を検討 |

## 7. 成功基準

### 最小成功（Must）
- `mogura ~` でホームディレクトリの使用量が詳細にわかる
- macOS 標準のストレージ管理より具体的な情報が得られる

### 目標成功（Should）
- キャッシュ・不要ファイルの検出で実際にディスクを節約できる
- JSON 出力で他ツールとの連携が可能

### 理想成功（Could）
- 並行スキャンで大規模ディレクトリも高速に分析できる
